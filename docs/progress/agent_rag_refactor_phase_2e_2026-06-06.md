# Agent-RAG 重构 Phase 2E 进度

日期：2026-06-06

## 本阶段目标

为论文派生资产增加不可变版本，并让论文范围审计绑定创建时的活动版本，消除论文重导入对历史运行证据集合的影响。

## 已完成内容

1. 不可变论文版本
   - 新增 `paper_versions`、`paper_active_versions`、`paper_version_sections` 和 `paper_version_evidence_blocks`。
   - 版本 ID 由论文 ID、标题和 Markdown 内容确定性生成。
   - 相同内容重导入复用已有版本。
   - 内容变化时创建新版本，并仅更新活动版本指针与兼容当前视图。
   - 历史版本章节与证据块只追加、不替换。

2. 版本绑定审计
   - 论文范围审计运行输入新增 `paper_version_id`。
   - 创建运行时快照活动版本和有序证据块 ID。
   - Worker 优先从不可变版本解析证据。
   - 保留 Phase 2D 当前视图快照和旧内联证据运行的兼容路径。

3. 旧数据升级
   - Repository 初始化时检测没有活动版本指针的旧论文。
   - 将旧当前视图章节和证据块回填为首个不可变版本。
   - 不修改旧运行输入，不对历史运行做不可验证的追溯绑定。

4. 版本检查 API
   - `GET /api/papers/{paper_id}/versions`
   - `GET /api/papers/{paper_id}/versions/{version_id}/evidence-blocks`
   - 当前论文元数据包含 `active_version_id`。

## 分层边界

- 当前视图表继续服务现有论文读取接口。
- 不可变版本表承担历史审计证据和可复现性。
- `PaperService` 负责确定性版本生成。
- `ReviewAuditService` 负责绑定活动版本。
- Worker 负责按运行输入中的版本 ID 精确解析证据。

## 当前限制与风险

- 不可变版本会持续占用本地 SQLite 存储，尚未实现保留策略和归档。
- 旧 Phase 2D 运行没有 `paper_version_id`，仍按原兼容路径执行。
- 当前版本哈希基于原始 Markdown；只改变空白也会形成新版本。
- 旧数据回填时没有原始 Markdown，只能基于当前派生资产生成版本 ID；未来重新导入同内容可能形成一个新的正式版本。
- 当前仍为 `silver diagnostic`，不替代人工金标评估。
- 未实现前端、PDF/MinerU 导入、鉴权、多租户和对象存储。

## 验证结果

- API 测试 18 项通过。
- 后端测试 31 项通过。
- 核心测试 25 项通过。
- Redis/RQ 显式集成测试 1 项通过。
- 真实版本漂移链路：
  - Worker 停止时导入版本 A 并创建论文范围审计。
  - 运行输入绑定版本 A，状态保持 queued。
  - 重导入版本 B 后活动版本发生变化，历史版本数量为 2。
  - Worker 启动后运行成功完成，未受活动版本 B 影响。
  - 版本 A 与版本 B 的历史证据接口分别返回对应内容。
  - 报告生成和 Markdown 读取成功。
- 旧库升级回归覆盖无版本论文资产自动回填。
- 自动化验证覆盖相同内容版本复用、变更内容新版本、历史证据不变、版本绑定审计和旧运行兼容。

## 下一阶段建议

1. 建立实验清单实体，将数据集版本、论文版本、运行配置、运行和报告关联。
2. 增加批量论文审计任务，支持实验复跑与论文结果表格生成。
3. 增加版本存储统计、归档和清理策略，但禁止删除仍被运行引用的版本。
