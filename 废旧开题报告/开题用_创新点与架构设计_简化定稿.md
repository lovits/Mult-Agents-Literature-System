# 开题用简化定稿：多Agent + RAG学术论文自动评审与分类系统

## 1. 课题定位

建议题目：

> 基于证据验证的多智能体 RAG 学术论文自动评审与接收倾向分类研究

一句话概括：

> 本课题在 MARG 多智能体评审生成框架基础上，加入评审意见级 RAG 证据验证与元评审排序模块，对每条候选评审意见进行“是否有证据、是否具体、是否已被论文解决、是否值得进入最终评审”的判断，从而提升自动评审意见的精准率、事实性、具体性和可操作性，并进一步用证据化评审特征辅助 Accept/Reject 分类。

本课题不建议做成庞大的“全功能 AI 审稿平台”。硕士论文最稳妥的路线是：

```text
MARG 高召回生成候选评审意见
        ↓
RAG Evidence Verifier 逐条证据验证
        ↓
Meta-Reviewer Ranker 去重、排序、置信度标注
        ↓
结构化评审报告 + 可选接收倾向分类
```

这样 baseline 明确、改进点集中、数据和指标都容易落地。

## 2. 本地论文分析结论

### 2.1 最适合做主 baseline：MARG

MARG 是本课题最合适的主 baseline，理由是架构简单、代码公开、数据和指标清楚。它用多个 LLM agent 分别从实验、清晰度、新颖性/影响力等角度生成评审意见，再做 refinement。

MARG 的关键结果显示，它的 MARG-S 版本 Recall 最高，但 Precision 和 Jaccard 不高，且评论数量很多：

| 方法 | Recall | Precision | Jaccard | 平均评论数 |
|---|---:|---:|---:|---:|
| SARG-B | 7.43 | 1.40 | 1.25 | 19.7 |
| SARG-TP | 10.62 | 4.61 | 3.46 | 11.6 |
| MARG-TP | 8.49 | 5.34 | 3.52 | 8.5 |
| LiZCa | 9.67 | 9.96 | 5.58 | 4.0 |
| MARG-S | 15.84 | 4.41 | 3.53 | 19.8 |
| Human | 9.42 | 12.00 | 5.45 | 4.7 |

这说明 MARG-S 的优势是“能找出更多可能的问题”，缺点是“候选意见太多，精准率不高”。这正好适合加一个后处理过滤器：保留高召回优势，同时提升 precision。

MARG 的 specificity 也有明确改进空间。MARG-S 已经比早期 baseline 更具体，约 71% 的评论被评为 specific/very specific，但仍有约 29% generic/very generic。对硕士论文来说，这个缺陷非常适合转化为可量化目标：

- Generic Rate 降低；
- Precision 提升；
- Top-K 核心意见有效率提升；
- Evidence Coverage 提升。

### 2.2 RAG 与证据验证的理论依据：FactReview、OpenNovelty、NoveltyAgent

FactReview 说明自动评审不应该只生成一段文字，而应该把判断拆成 claim-level evidence assessment。它对每个主要声明输出 Supported、Supported by the paper、Partially supported、In conflict、Inconclusive 等标签，并把判断链接到论文、相关文献或执行证据。它的价值不是作为主 baseline，而是支撑本课题的核心思想：评审意见必须能被证据检查。

OpenNovelty 和 NoveltyAgent 都强调 point-wise novelty analysis。它们把论文贡献点拆成多个 novelty claim，再检索相关工作做逐点比较。对本课题的启发是：RAG 不能只检索一堆相关文献，而应该围绕“评审意见或贡献点”进行细粒度检索和验证。

因此，本课题的 RAG 模块应定位为：

> 面向评审意见的证据验证，而不是普通的相关文献增强。

### 2.3 强但不适合做主 baseline：ReviewGrounder、ScholarPeer

ReviewGrounder 和 ScholarPeer 都非常前沿，但不建议作为主 baseline。

ReviewGrounder 有完整的 rubric-guided evaluation，指标体系很强，还明确提到未来可以探索 LLM-based meta-reviewer 和 iterative feedback loop。它适合放在相关工作和评价指标参考中，也可以作为后期强 baseline；但它本身已经复杂，复现成本高于 MARG。

ScholarPeer 是 search-enabled multi-agent framework，包含 historian agent、baseline scout agent、Q&A agent，能解决“真空评审”问题。它和本课题方向很接近，但工程依赖较重，且评估依赖 search-enabled judge 和较新模型。开题时可以把它作为“强相关竞品”，但不建议把论文实现建立在它上面。

### 2.4 分类任务的数据依据：PeerRead、DeepReview-13K

如果论文题目里保留“分类”，建议把分类作为第二实验，而不是主任务。

PeerRead 是经典公开数据集，包含 14.7K 篇论文草稿、accept/reject 决策，以及子集中的 10.7K 条专家评审。它适合做 accept/reject 分类或 review-aware 分类。

DeepReview-13K 更贴近当前 LLM 评审场景，包含 ICLR 2024/2025 投稿、评分、接收决策和结构化 reasoning。它提供 Decision Accuracy、Decision F1、Rating MSE/MAE、Spearman 等指标，适合做扩展验证。

### 2.5 评审质量辅助评估：RottenReviews

RottenReviews 提供超过 15,000 个投稿/评审数据，以及 700 多个人工标注 paper-review pairs，覆盖 13 个可解释的评审质量维度，包括 Factuality、Vagueness、Actionability、Overall Quality 等。

这与本课题目标高度一致。主实验仍然建议跟 MARG 的 Precision/Recall/Jaccard 对齐；辅助实验可以用 RottenReviews 的 Factuality、Specificity/Vagueness、Actionability 证明输出质量更好。

## 3. 研究空白

本课题可以把研究空白写成三点。

第一，现有多智能体评审系统重视“生成更多评审意见”，但缺少评审意见级证据验证。MARG 的高 recall 说明多 agent 能提出很多候选问题，但 precision 偏低、评论数偏多，说明系统缺少对候选意见的验证和筛选。

第二，已有 RAG/证据核查工作更多用于论文 claim 或 novelty claim，而不是直接用于过滤自动生成的 review comments。FactReview、OpenNovelty、NoveltyAgent 都证明了证据化判断的重要性，但它们不是针对 MARG 这类 review generation 输出做后处理。

第三，论文接收倾向分类通常直接基于全文或评论文本预测，缺少可解释的中间特征。本课题可以把“有效 major weakness 数量、证据覆盖率、无效意见率、置信度”等作为分类特征，让分类结果更可解释。

## 4. Baseline 选择

### 4.1 主 baseline

主 baseline：MARG-S。

选择原因：

| 维度 | 判断 |
|---|---|
| 架构简单 | 多 agent prompt 流程，不需要训练大模型 |
| 数据好找 | ARIES 数据与复现流程相对明确 |
| 指标清楚 | Recall、Precision、Jaccard、评论数、Specificity |
| 缺陷明确 | 高 recall 但 precision 低，评论数量多，仍有 generic comments |
| 改进自然 | 加 RAG Evidence Verifier 属于低风险后处理，不破坏原系统 |

### 4.2 对比 baseline

建议设置以下对比：

| 方法 | 用途 |
|---|---|
| GPT-4/强 LLM 直接评审 | 单模型无 agent baseline |
| MARG-S | 主 baseline |
| MARG-S + LLM-only Filter | 验证“纯 LLM 判断”是否足够 |
| MARG-S + Paper-RAG Verifier | 验证论文内证据检索的效果 |
| MARG-S + Paper-RAG + Literature-RAG | 验证外部文献证据是否进一步提升 |
| MARG-S + RAG Verifier + Meta-Ranker | 本文完整方法 |

ReviewGrounder/ScholarPeer 可以作为强相关工作讨论，若时间充足再做小规模对比，不建议作为开题阶段承诺的主复现对象。

## 5. 创新点设计

### 创新点一：评审意见级 RAG Evidence Verifier

核心问题：

> MARG 生成了很多候选评审意见，但没有逐条判断这些意见是否真的被论文或相关文献支持。

改进方法：

对 MARG 输出的每条 comment，构造查询并检索论文全文和相关文献，然后判断该 comment 属于哪一类：

| 标签 | 含义 | 处理 |
|---|---|---|
| Valid-Supported | 有论文内证据或外部文献证据支撑 | 保留 |
| Invalid-Covered | 评论指出的问题其实论文已经解决 | 过滤 |
| Invalid-Hallucinated | 评论提到的内容在论文中不存在或与论文矛盾 | 过滤 |
| Generic | 太泛化，缺少具体 section/table/dataset/metric | 降权或重写 |
| Weak-Evidence | 可能有价值但证据不足 | 标记需人工复核 |

与普通 RAG 的区别：

普通 RAG 是“给 LLM 更多上下文生成评审”，本课题是“对已经生成的评审意见做证据审计”。这使创新点更具体，也更容易评估。

量化指标：

| 指标 | 定义 | 预期方向 |
|---|---|---|
| Precision | 保留意见中与人工评审对齐的比例 | 提升 |
| Recall | 覆盖人工评审意见的比例 | 可小幅下降 |
| Jaccard | 生成意见集合与人工意见集合重叠 | 提升 |
| Invalid Rate | 已解决、幻觉、无证据意见比例 | 下降 |
| Evidence Coverage | 带 section/table/文献证据的意见比例 | 提升 |
| Generic Rate | 泛化意见比例 | 下降 |

这是论文的主创新点。

### 创新点二：证据感知 Meta-Reviewer Ranker

核心问题：

过滤以后仍然可能存在重复意见、轻重不分、置信度不清的问题。真实审稿中，area chair/meta-reviewer 会把多个 reviewer 的意见整合成少数核心问题。

改进方法：

设计一个轻量 Meta-Reviewer Ranker，不重新生成大段评审，而是做四件事：

1. 去重：合并语义相近的评论；
2. 排序：按照 `严重性 × 证据强度 × 可操作性 × 新颖性影响` 给评论排序；
3. 置信度标注：区分 high-confidence concern 和 needs-human-check；
4. 结构化输出：生成 Strengths、Major Weaknesses、Minor Weaknesses、Questions、Evidence Anchors。

推荐打分公式：

```text
score(comment) =
  0.35 * severity
+ 0.30 * evidence_strength
+ 0.20 * actionability
+ 0.15 * novelty_or_decision_impact
```

量化指标：

| 指标 | 定义 |
|---|---|
| Top-K Precision | Top-3/Top-5 major concerns 中有效意见比例 |
| Redundancy Rate | 重复或高度相似评论比例 |
| Actionability | 是否给出明确可执行修改建议 |
| Confidence Calibration | 置信度与人工有效性标签是否一致，可用 ECE/Brier |

### 创新点三：证据化评审特征辅助接收倾向分类

分类不是主创新，但可以作为第二实验增强论文完整性。

核心思想：

不用只拿论文全文直接预测 accept/reject，而是使用经过验证的评审特征：

```text
features =
  valid_major_weakness_count,
  valid_strength_count,
  evidence_coverage,
  invalid_rate,
  top3_concern_confidence,
  novelty_risk_score,
  reproducibility_risk_score,
  experiment_weakness_score,
  clarity_weakness_score
```

分类模型可以很简单：

- Logistic Regression；
- XGBoost/LightGBM；
- 小型 MLP；
- 或 LLM-based classifier 作为对比。

量化指标：

| 数据集 | 指标 |
|---|---|
| PeerRead | Accuracy、Macro-F1、AUC |
| DeepReview-13K | Decision Accuracy、Decision F1、Rating MSE/MAE、Spearman |

实验目标：

证明“证据化评审特征”比直接使用原始 MARG 评论或全文 embedding 更可解释，且分类效果不低于 baseline。

## 6. 简化系统架构

### 6.1 总体架构

```text
输入论文 PDF
   │
   ▼
Stage 0: Paper Parser & Indexer
   - PDF/text 解析
   - section/paragraph/table caption 切分
   - 构建 BM25 + 向量索引
   - 抽取 title、abstract、claims、datasets、baselines、metrics
   │
   ▼
Stage 1: MARG-S Candidate Review Generation
   - Experiment Reviewer
   - Clarity/Reproducibility Reviewer
   - Novelty/Impact Reviewer
   - Refinement Agent
   输出候选评论 C = {c1, c2, ..., cn}
   │
   ▼
Stage 2: Evidence-aware RAG Verifier
   对每条 ci：
   - query rewriting
   - Paper-RAG 检索论文内证据
   - Literature-RAG 检索相关工作证据，可选
   - Evidence Judge 输出标签、证据、置信度
   │
   ▼
Stage 3: Meta-Reviewer Ranker
   - 过滤无效意见
   - 合并重复意见
   - 排序 Top-K major concerns
   - 标注 confidence / needs-human-check
   │
   ├──► 输出结构化评审报告
   │
   └──► Optional Classification Head
          用证据化评审特征预测 Accept/Reject
```

### 6.2 模块职责

| 模块 | 职责 | 是否必须 |
|---|---|---|
| Paper Parser | 解析 PDF/文本，切分段落和章节 | 必须 |
| Paper Indexer | 构建 BM25/向量索引 | 必须 |
| MARG-S Generator | 生成候选评审意见 | 必须 |
| Query Rewriter | 把 comment 改写成检索查询 | 必须 |
| Paper-RAG Retriever | 检索论文内相关证据 | 必须 |
| Literature-RAG Retriever | 检索外部相关文献 | 可选增强 |
| Evidence Judge | 判断 comment 是否有效 | 必须 |
| Meta-Reviewer Ranker | 去重、排序、置信度标注 | 必须 |
| Classifier | 接收倾向分类 | 可选第二实验 |

### 6.3 为什么架构简单

本方案不训练大模型，也不重写 MARG 内部多 agent 流程，而是在 MARG 输出后增加两个轻量模块：

1. Evidence Verifier；
2. Meta-Reviewer Ranker。

主要工程量是：

- 文本解析；
- 检索索引；
- prompt 设计；
- 标签输出解析；
- 指标计算。

这些都适合硕士阶段实现。

## 8. 实验设计

### 8.1 主实验

目标：证明加入 RAG Evidence Verifier 后，MARG 输出更精准、更具体、更有证据。

| 方法 | 说明 |
|---|---|
| GPT-4 Direct Review | 单模型直接评审 |
| MARG-S | 主 baseline |
| MARG-S + LLM-only Filter | 不检索，只让 LLM 判断评论有效性 |
| MARG-S + Paper-RAG Verifier | 检索论文内证据 |
| MARG-S + Paper-RAG + Literature-RAG | 加外部文献证据 |
| Ours Full | Verifier + Meta-Ranker |

比较指标：

- Precision；
- Recall；
- Jaccard；
- Generic Rate；
- Evidence Coverage；
- Top-K Precision。

### 8.2 消融实验

| 消融 | 目的 |
|---|---|
| 去掉 Paper-RAG | 验证论文内证据检索是否必要 |
| 去掉 Literature-RAG | 验证外部文献证据的边际收益 |
| 去掉 Meta-Ranker | 验证排序和去重是否有效 |
| BM25 vs Dense vs Hybrid | 验证检索策略 |
| keep/filter vs keep/filter/revise | 验证是否需要重写低质量意见 |

### 8.3 分类扩展实验

目标：证明证据化评审特征对 accept/reject 分类有帮助。

| 方法 | 特征 |
|---|---|
| Text-only baseline | 论文 title/abstract/full text embedding |
| Review-only baseline | MARG 原始评论 embedding |
| Ours | 证据验证后的结构化评审特征 |
| Ours + Text | 结构化特征 + 论文文本特征 |

比较指标：

- Accuracy；
- Macro-F1；
- AUC；
- 可解释性案例分析。

## 9. 开题报告中可直接使用的创新点表述

### 创新点 1：评审意见级证据验证机制

针对现有多智能体评审系统生成意见多但精准率不足的问题，本文提出评审意见级 RAG Evidence Verifier。该模块以自动生成的每条评审意见为验证对象，检索论文原文段落和相关文献证据，判断该意见是否被证据支持、是否已经被论文解决、是否存在幻觉或过度泛化。与传统 RAG 直接增强生成不同，本文将 RAG 用于评审意见的后验审计和有效性过滤，从而提升评审意见的 precision、factuality 和 evidence coverage。

### 创新点 2：证据感知元评审排序机制

针对自动评审输出中重复、轻重不分、置信度不清的问题，本文设计证据感知 Meta-Reviewer Ranker，对候选评审意见进行去重合并、重要性排序和置信度标注。该模块综合考虑问题严重性、证据强度、可操作性和决策影响，输出 Top-K 核心问题及其证据锚点，使最终评审报告更接近人类元评审的组织方式。

### 创新点 3：面向接收倾向分类的证据化评审特征

本文进一步将证据验证后的评审结果转化为可解释分类特征，包括有效主要缺陷数量、证据覆盖率、无效意见率、实验风险、新颖性风险和置信度等，用于辅助论文 Accept/Reject 分类。相比直接使用论文全文或原始生成评论，该方法提供了更可解释的中间决策依据。

## 10. 最终推荐答辩话术

如果导师问“你的 baseline 是什么”，可以回答：

> 我的主 baseline 是 MARG。MARG 是一个公开的多智能体论文评审生成框架，优点是 recall 高，能生成较多候选评审意见；但它的 precision 较低，MARG-S 的 Precision 只有 4.41，且平均生成 19.8 条评论，说明存在评论过多、无效或泛化的问题。因此我不是从零设计一个复杂系统，而是在 MARG 的候选评审意见之后增加证据验证和元评审排序模块。

如果导师问“你的创新点是什么”，可以回答：

> 我的核心创新不是简单地把 RAG 接到 LLM 前面，而是把 RAG 用在评审意见级的后验验证上。系统会对每条自动生成的评审意见检索论文内证据和外部文献证据，判断它是否有效、是否已被论文解决、是否泛化或幻觉，然后由元评审排序模块输出 Top-K 核心问题。这可以直接用 Precision、Invalid Rate、Evidence Coverage、Top-K Precision 等指标量化。

如果导师问“为什么这个能做出来”，可以回答：

> 这个方案工程上比较可控。MARG 提供了公开 baseline 和现成评估指标；RAG Evidence Verifier 是后处理模块，不需要训练大模型；分类部分可以用 PeerRead 或 DeepReview-13K 做扩展实验；评审质量可以用 RottenReviews 的 Factuality、Actionability、Vagueness 等维度做辅助验证。因此它既有创新点，也有明确可复现实验。

## 11. 建议删减的高风险内容

以下内容可以放在未来工作，不建议作为主承诺：

| 内容 | 原因 |
|---|---|
| 完整 6-Agent 大系统 | 工程量大，创新点分散 |
| 代码执行式 claim verification | 类似 FactReview，环境复现复杂 |
| 动态 web-scale search | 依赖搜索 API 和最新模型，稳定性差 |
| 中文论文评审适配 | 额外数据和评估成本高 |
| 训练/RL 优化自动评审模型 | GPU 和数据成本高 |
| 端到端替代人类审稿 | 表述风险高，伦理上不合适 |

更稳妥的表述是：

> 本系统定位为人类审稿人的辅助工具，用于提高候选评审意见的证据性、具体性和可操作性，而不是替代最终审稿决策。
