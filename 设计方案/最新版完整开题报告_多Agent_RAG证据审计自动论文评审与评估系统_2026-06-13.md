# 开题报告：基于双向证据审计与双 RAG 的多智能体学术论文自动评审与评估系统研究

版本：2026-06-13
系统名称：EviReview-Lite
研究类型：计算机科学与技术、软件工程、人工智能方向的工程创新型硕士毕业设计

## 摘要

随着学术论文投稿数量持续增长，人工同行评审面临审稿负担加重、评审深度不稳定和意见证据不足等问题。大语言模型能够生成结构完整、语言流畅的论文评审，但已有研究表明，其输出仍容易包含泛化批评、论文误读、已覆盖问题和缺乏依据的意见。现有多智能体评审方法主要提升候选意见的覆盖度与具体性，但通常缺少对每条评审意见是否成立的细粒度验证；现有 RAG 与科学事实核验方法则较少直接面向自动评审意见的过滤、排序与报告生成。

本课题拟设计并实现 EviReview-Lite，一套基于多智能体与双 RAG 的自动论文评审与评估系统。系统首先由方法、实验和可复现性评审 Agent 生成原子化候选弱点，再由 Query Planner 为每条弱点声明目标章节、预期证据类型和检索查询。Paper-RAG 使用 BM25、Dense Retrieval、RRF、章节先验、证据类型先验和邻近块扩展检索论文内部证据；受控 Literature-RAG 针对新颖性、遗漏 baseline、相关工作和外部比较类意见检索外部文献证据。随后，Support Agent 与 Refutation Agent 分别构建支持和反驳案例，Evidence Adjudicator 自动输出 `keep`、`rewrite`、`reject` 或 `uncertain` 裁决，Meta-Reviewer 根据证据充分性、有效概率、严重性、具体性、可操作性和重复度生成 Top-K 结构化评审报告。

课题以工程创新和可复现实验为重点，不训练大模型，不进行大规模人工标注，不设置人工复核流程，也不把自动接收或拒绝决策作为主要目标。实验采用 CLAIMCHECK、PeerQA、OpenReview、SubstanReview、RottenReviews 和本地受控文献语料，设置 Direct LLM、Structured Prompt、MARG-lite、普通 Hybrid RAG、单 Judge 验证和基础排序方法等 Baseline，从检索、裁决、排序和端到端四个层面评价系统。预期成果包括可运行的后端 Agent-RAG 系统、可复现实验流水线、组件级消融实验、结构化审计轨迹和论文实验报告。

**关键词：** 多智能体；检索增强生成；自动论文评审；证据审计；学术论文评估；元评审

## 一、课题名称与研究定位

### 1.1 推荐课题名称

**基于双向证据审计与双 RAG 的多智能体学术论文自动评审与评估系统研究**

### 1.2 课题定位

本课题属于偏工程创新方向的硕士毕业设计。研究重点不是训练新的基础模型，也不是模拟完整会议录用流程，而是设计一条可复现、可解释、可量化的自动评审后端流水线：

```text
候选弱点生成
  -> 论文内部与外部证据检索
  -> 支持/反驳双向审计
  -> 自动裁决
  -> 证据感知去重与 Top-K 排序
  -> 带证据锚点的结构化自动评审报告
```

### 1.3 应用场景

系统主要面向：

1. 作者投稿前的自动论文自检；
2. 审稿人阅读论文时的问题定位辅助；
3. 自动评审模型的组件级实验与可靠性评价；
4. Agent-RAG 在长篇学术文档分析场景中的工程研究。

系统不被定义为正式同行评审的替代者，不输出最终录用决策。低置信度或证据冲突无法消解的意见自动标记为 `uncertain`，保留在审计日志中但不进入最终 Top-K 结论。

## 二、研究背景与意义

### 2.1 研究背景

同行评审需要审稿人理解论文方法、实验、贡献和相关工作，并判断论文论证是否充分。随着投稿规模增加，审稿人在有限时间内完成深入阅读和证据核对的难度持续增加。公开同行评审数据和大语言模型的发展，使自动论文评审成为可研究的工程问题。

已有工作证明 LLM 可以提供有用的论文反馈，但直接生成的评审常存在具体性不足、错误批评和证据缺失 [25]。MARG 通过多个专长 Agent 提升评审覆盖度和具体性，其用户研究中通用评论比例由 60% 降至 29%，但该方法仍主要解决“如何生成更多、更具体的反馈”，不能保证每条反馈成立 [1]。CLAIMCHECK 进一步表明，评审意见是否与论文 claim 对齐、是否客观和有效，需要专门的数据与评价方法 [2]。

近两年的研究方向正在从“生成完整 review”转向“结构化调查、证据定位和可审计输出”。TreeReview 使用问题树分解评审任务 [3]；DeepReviewer 2.0 构建可追踪的 claim-evidence-risk ledger [4]；ProReviewer 使用结构化 review log 主动调查论文 [5]；FactReview 和 ReviewGrounder 强调外部证据、工具调用和 grounded critique [6][7]。这些工作说明，自动评审系统的关键竞争点正在转向证据性、可追踪性和可靠性。

### 2.2 理论与方法意义

本课题将自动评审从一次性文本生成重新定义为可分解的证据审计任务：

1. 将评审意见作为独立、可验证的原子对象；
2. 将检索质量、意见有效性和最终报告质量分开评价；
3. 将支持证据与反驳证据同时纳入自动裁决；
4. 将论文结构和证据类型显式加入 RAG；
5. 将元评审设计为可解释的过滤、去重和排序过程。

### 2.3 工程实践意义

本课题能够形成一套完整的 Agent-RAG 后端工程：

- 支持论文解析、索引、检索、Agent 编排和结果追踪；
- 每条最终意见均可追溯至论文段落、表格、附录或外部文献；
- 各组件具备独立 Baseline 和消融实验；
- 系统可在无需模型训练和大规模人工标注的条件下完成；
- 研究结果可以直接转化为毕业论文中的系统设计、实验和误差分析章节。

## 三、国内外研究现状与文献综述

### 3.1 自动论文评审数据与早期任务

PeerRead 提供论文草稿、专家评审和接收决策，是早期论文接收预测与评审分析的重要数据集 [8]。ASAP-Review 将论文评审生成和评价拆分为完整性、合理性、依据性等维度 [9]。OpenReview 公开了投稿、评审、回复和讨论等 Note 对象，为当前自动评审研究提供了真实数据基础 [10]。

这些资源解决了“是否有数据”的问题，但直接用接收标签评价自动评审容易混淆论文质量、审稿偏差和会议录用机制。因此，本课题不把接收预测作为主要实验，而优先使用评审弱点、问题回答、证据定位和评审质量标签。

### 3.2 LLM 与多智能体自动评审

MARG 使用多个 LLM 实例分配论文片段，并设置实验、清晰度和影响力等专家角色，提高评审意见的具体性和覆盖度 [1]。AgentReview 使用 Reviewer、Author 和 Area Chair 等角色模拟同行评审动态，主要研究不同角色、偏差和机制对评审过程的影响 [11]。DeepReview 使用新颖性验证、多维评审和可靠性验证构建结构化评审流程及 DeepReview-13K 数据 [12]。TreeReview 通过动态问题树对评审问题进行逐层分解，并报告了相对 MARG 的精度和 token 效率提升 [3]。

上述方法证明多 Agent 分工和结构化推理有价值，但完整评审生成仍可能保留误读、重复和缺乏证据的意见。因此，本课题把多 Agent 评审器限定为候选弱点生成器，并在生成后设置独立证据审计。

### 3.3 证据校验与可审计评审

SubstanReview 将评审质量中的 substantiation 建模为评审文本中的 claim-evidence pair 抽取，并提供 550 条专家标注评审 [13]。CLAIMCHECK 针对 NeurIPS 2023 和 2024 投稿与评审，标注评审弱点、被质疑的论文 claim，以及弱点有效性、客观性和类型，为弱点级验证提供了直接数据基础 [2]。

RefChecker 使用细粒度 claim-triplet 检测生成内容中的 entailment、neutral 和 contradiction [14]。RAGChecker 将 RAG 评价拆分为检索器、生成器和整体指标，并验证了细粒度指标与人类判断的相关性 [15]。这些工作支持本课题把自动评审拆分为“证据检索是否正确”和“裁决是否正确”两个独立实验。

FactReview 将论文主要 claim 与外部文献、论文内容和可执行代码结果对齐，并输出 Supported、Partially Supported、In Conflict 和 Inconclusive 等证据标签 [6]。DeepReviewer 2.0、ProReviewer 和 EGTR-Review 进一步强调审计轨迹、证据状态和可追踪输出 [4][5][16]。但代码执行、强化学习和教师蒸馏会显著扩大工作量，本课题只借鉴其证据轨迹与输出约束思想。

### 3.4 论文结构感知与 Literature-RAG

PeerQA 包含来自同行评审问题的 579 个 QA 对，覆盖 208 篇论文，答案由原论文作者标注，适合评价论文内部证据检索 [17]。MA-RAG 说明 Planner、Extractor 和 QA 等专门 Agent 可以形成可解释、可消融的模块化 RAG 流程 [24]。ScholarPeer 使用 historian、baseline scout 和多维 Q&A Agent 获取外部领域背景并发现缺失比较 [18]。OpenNovelty 和 NoveltyAgent 将论文贡献拆成细粒度 novelty points，进行相关工作检索、全文比较和引用验证 [19][20]。

上述研究说明，论文内部事实核验与外部文献比较是两种不同检索任务。普通全文 RAG 难以同时处理“论文是否已经覆盖该问题”和“外部是否存在遗漏 baseline”。因此，本课题采用职责分离的 Paper-RAG 与 Literature-RAG。

### 3.5 自动评审评价体系

RottenReviews 收集多个开放评审来源，并提供超过 700 个具有人类多维质量标注的 paper-review 对。其结果显示，零样本和微调 LLM 与专家评价的对齐仍有限，简单可解释特征模型可能优于 LLM judge [21]。PRISM 从分析深度、新颖性评估、缺陷发现与优先级、建设性四个维度评价 LLM reviewer，并指出不同系统存在不同能力盲点 [22]。MMReview 则扩展到多学科和多模态评审场景 [23]。

因此，本课题不使用单一 ROUGE、单一接收预测或单一 LLM-as-Judge 作为成功标准，而采用 gold 标签、检索指标、裁决指标、Top-K 指标、成本指标和案例审计相结合的评价方案。

### 3.6 现有研究不足

综合现有研究，可归纳出以下研究空白：

1. 多智能体方法擅长扩展候选意见，但对错误意见、已覆盖问题和重复意见缺少稳定过滤机制；
2. 现有 grounded review 多以单向证据支持为主，较少为每条候选意见同时构建支持与反驳案例；
3. 普通 RAG 很少显式建模论文章节与证据类型之间的对应关系；
4. 外部文献检索常被设计为大规模在线系统，不适合作为硕士工程实验的核心依赖；
5. 自动评审评价常只关注最终文本，缺少组件级、可诊断、可复现的成功标准；
6. 最新复杂系统依赖强化学习、微调、蒸馏或大规模语料，复现成本较高。

## 四、研究问题、目标与范围

### 4.1 研究问题

**RQ1：** 结构与证据类型感知的 Paper-RAG 能否比 BM25、Dense 和普通 Hybrid RAG 更准确地定位评审意见所需的论文内部证据？

**RQ2：** 对每条候选弱点固定构建 Support 与 Refutation 案例，能否比 LLM-only Judge 和单 Judge + Paper-RAG 更准确地过滤不成立、已覆盖或被反驳的意见？

**RQ3：** 受控 Literature-RAG 能否为新颖性、遗漏 baseline 和相关工作类意见提供有效、可追溯的外部证据？

**RQ4：** 证据驱动的 Meta-Reviewer 能否在保持主要弱点覆盖率的同时，提高 Top-K 精度并降低重复率和错误保留率？

**RQ5：** 完整 EviReview-Lite 是否能够比 Direct LLM、Structured Prompt 和 MARG-lite 输出更具体、更有证据且错误更少的自动评审报告？

### 4.2 研究目标

1. 构建一套分层清晰、可运行的多 Agent-RAG 自动论文评审后端；
2. 实现结构与证据类型感知的 Paper-RAG；
3. 实现小规模、可复现的 Literature-RAG；
4. 实现固定 Support、Refutation 与 Adjudicator 双向审计协议；
5. 实现证据感知的弱点过滤、论文内去重和 Top-K 排序；
6. 建立统一数据协议、实验追踪、消融和统计检验流程；
7. 使用公开数据集完成组件级和端到端实验。

### 4.3 研究边界

#### 必做范围

- 8 个固定职责 Agent；
- Paper-RAG 与受控 Literature-RAG；
- 双向证据审计；
- Meta-Reviewer Top-K 排序；
- E0–E6 实验与主要消融；
- 自动生成带证据锚点的结构化评审。

#### 不做范围

- 不训练、微调或强化学习大模型；
- 不做大规模在线文献爬取；
- 不执行论文代码以复现实验结果；
- 不进行多模态图像内容理解主实验；
- 不进行大规模新增人工标注；
- 不设置人工复核或人工决策节点；
- 不将 accept/reject 分类作为论文主创新或主要成功标准。

## 五、拟解决的关键问题与创新点

### 5.1 创新点一：评审意见级双向证据审计

现有自动评审常使用单个 Judge 判断意见是否合理，容易受到检索偏差和确认偏差影响。本课题针对每条原子化候选弱点固定构建两类案例：

```text
Candidate Weakness
  -> Support Case：证明问题成立
  -> Refutation Case：证明问题已覆盖、被误读或不成立
  -> Evidence Adjudicator
  -> keep / rewrite / reject / uncertain
```

该机制的工程创新在于统一输入输出协议、固定执行路径和完整审计轨迹。`uncertain` 表示系统自动拒绝形成确定结论，不进入最终 Top-K。

### 5.2 创新点二：结构与证据类型感知的双 RAG

Paper-RAG 不仅根据语义相似度检索，还根据弱点类型声明：

```json
{
  "aspect": "experiment",
  "target": "retrieval module",
  "expected_sections": ["Experiments", "Ablation", "Appendix"],
  "expected_evidence_types": ["table", "paragraph", "implementation_detail"],
  "queries": ["retrieval module ablation", "remove retriever component"]
}
```

Paper-RAG 负责论文内部证据；Literature-RAG 只对 `novelty`、`related_work`、`missing_baseline` 和 `external_comparison` 四类固定任务运行。该设计不是争议触发升级，而是按任务类型确定的固定职责边界。

### 5.3 创新点三：双 RAG 证据驱动的 Meta-Reviewer Ranker

Meta-Reviewer 不重新生成大量新问题，而是对经过审计的弱点执行：

1. 删除 `reject` 和 `uncertain`；
2. 自动收窄 `rewrite`；
3. 合并同一论文内语义和证据范围重复的意见；
4. 根据证据充分性、有效概率、严重性、具体性和可操作性排序；
5. 输出 Top-3/Top-5 核心问题、置信度和证据锚点。

推荐排序公式：

```text
rank_score =
  0.30 * evidence_sufficiency
  + 0.20 * validity_probability
  + 0.20 * severity
  + 0.15 * specificity
  + 0.10 * actionability
  + 0.05 * literature_support
  - redundancy_penalty
  - uncertainty_penalty
```

### 5.4 候选弱点生成的定位

多 Reviewer Agent 生成候选弱点是系统必要组成，但不是独立创新。其作用是建立高召回候选池，供后续审计和排序。该模块通过 Direct LLM、Structured Prompt 和 MARG-lite 进行对比。

## 六、总体 Agent-RAG 架构

### 6.1 总体流程

```text
Stage 0 论文解析与索引
  PDF / Markdown
  -> section / paragraph / table caption / appendix
  -> Paper Evidence Index

Stage 1 候选弱点生成
  Method Reviewer
  + Experiment Reviewer
  + Reproducibility Reviewer
  -> Atomic Candidate Weakness Pool

Stage 2 查询规划与双 RAG
  Query Planner
  -> Paper-RAG：所有候选弱点
  -> Literature-RAG：四类外部比较任务
  -> Evidence Bundle

Stage 3 双向证据审计
  Support Agent
  + Refutation Agent
  -> Evidence Adjudicator
  -> keep / rewrite / reject / uncertain

Stage 4 Meta-Reviewer
  -> 自动过滤与改写
  -> 论文内去重
  -> 证据感知排序
  -> Top-K 结构化评审报告
```

### 6.2 八个核心 Agent

| Agent | 输入 | 输出 | 核心职责 |
| --- | --- | --- | --- |
| Method Reviewer | 方法、算法与理论章节 | 方法类候选弱点 | 检查假设、推导、方法合理性 |
| Experiment Reviewer | 实验、表格、消融与结果 | 实验类候选弱点 | 检查 baseline、消融、指标与结论 |
| Reproducibility Reviewer | 实现细节、设置与附录 | 复现类候选弱点 | 检查超参数、流程和报告完整性 |
| Query Planner | 原子弱点与论文结构 | 双 RAG 查询计划 | 声明章节、证据类型和查询 |
| Support Agent | 弱点与证据包 | Support Case | 形成问题成立的证据链 |
| Refutation Agent | 弱点与证据包 | Refutation Case | 寻找已覆盖、误读或反例证据 |
| Evidence Adjudicator | 双向案例 | 自动裁决对象 | 输出动作、置信度和理由 |
| Meta-Reviewer | 全部裁决对象 | Top-K 评审报告 | 过滤、去重、排序和报告生成 |

### 6.3 普通模块

以下组件不包装成 Agent：

- Paper Parser；
- Section Normalizer；
- Evidence-Type Classifier；
- BM25、Dense Retriever、RRF 和可选 reranker；
- Candidate Normalizer；
- Deduplicator；
- Rank Score Calculator；
- Confidence Calculator；
- Metric Evaluator；
- Experiment Manifest Recorder。

这种划分避免为了“多 Agent”而增加不必要角色，使 Agent 只负责需要语言理解、规划和裁决的任务。

### 6.4 Paper-RAG

```text
Keyword Query + Semantic Query
  -> BM25 Top-K
  -> Dense Top-K
  -> RRF Fusion
  -> Section Prior
  -> Evidence-Type Prior
  -> Neighbor Expansion
  -> Paper Evidence Bundle
```

证据块类型包括：

- `paragraph`
- `table_caption`
- `figure_caption`
- `algorithm`
- `implementation_detail`
- `appendix`
- `absence_signal`

### 6.5 Literature-RAG

```text
Contribution / Baseline Query
  -> Local Full-text Corpus
  -> OpenAlex / Semantic Scholar Metadata
  -> BM25 + Dense
  -> Year / Topic / Citation Filter
  -> Literature Evidence Bundle
```

主实验使用固定快照和本地全文语料，在线搜索只用于扩展。输出必须包含标题、年份、来源、文献 ID 或 URL，以及与候选意见相关的证据片段。

### 6.6 审计对象协议

每条弱点保存：

```json
{
  "paper_id": "paper_001",
  "candidate_id": "w_001",
  "aspect": "experiment",
  "weakness": "The retrieval module lacks an ablation study.",
  "query_plan": {},
  "paper_evidence": [],
  "literature_evidence": [],
  "support_case": {},
  "refutation_case": {},
  "decision": "keep",
  "confidence": 0.84,
  "rank_features": {},
  "trace": []
}
```

## 七、后端工程架构与技术栈

### 7.1 架构风格

项目采用常规分层后端架构，并将 Agent 与 RAG 作为独立领域模块：

```text
controller/      API 与实验入口
service/         评审流程和实验用例编排
agent/           八个 Agent、Prompt 与输出解析
rag/             Paper-RAG、Literature-RAG、索引和检索
dao/             数据集、运行记录和索引元数据访问
models/          Pydantic 领域模型
conf/            模型、检索和实验配置
evaluation/      指标、统计检验和论文表格
scripts/         数据准备、索引、运行和复现脚本
tests/           单元、集成与回归测试
```

Controller-Service-DAO 分层负责后端工程边界；Agent-RAG 流程属于 Service 层管理的领域能力。该结构比传统 MVC 更适合后端实验系统，因为系统没有以页面视图为核心。

### 7.2 技术栈

| 类型 | 技术选型 | 选择理由 |
| --- | --- | --- |
| 开发语言 | Python 3.11 | 适合 LLM、检索与实验生态 |
| 后端接口 | FastAPI | 提供实验和评审服务接口 |
| 数据模型 | Pydantic | 强制 Agent 输入输出结构化 |
| Agent 编排 | LangGraph 或轻量状态机 | 固定流程、状态可追踪 |
| LLM 接口 | OpenAI-compatible Adapter | 解耦具体模型供应商 |
| 稀疏检索 | rank-bm25 或 Pyserini | 提供关键词精确检索 |
| 向量检索 | sentence-transformers + Qdrant/FAISS | 支持语义检索和本地实验 |
| 融合排序 | RRF + 可解释线性 Ranker | 易消融、易分析 |
| 论文解析 | MinerU / GROBID / 已有 Markdown | 保留章节、表格和附录结构 |
| 数据存储 | JSONL、Parquet、SQLite/PostgreSQL | 支持实验追踪和结构化查询 |
| 实验分析 | pandas、scipy、statsmodels | 指标、置信区间和显著性检验 |
| 测试 | pytest / unittest | 保证组件和流程稳定 |

### 7.3 运行条件

- CPU 可完成数据处理、BM25 和小规模实验；
- Dense embedding 推荐使用具备 8–16 GB 显存的 GPU，也可使用云端 embedding；
- LLM 使用兼容 API，通过环境变量配置，不在代码中保存密钥；
- Qdrant 可使用本地 Docker 或内存模式；
- 每次实验固定模型版本、温度、Prompt、随机种子、token 限制和索引快照。

## 八、数据源与数据方案

### 8.1 主数据源

| 数据源 | 可靠性依据 | 用途 | 新增人工标注 |
| --- | --- | --- | --- |
| CLAIMCHECK | EMNLP 2025 Findings；ML 专家标注弱点、论文 claim、有效性和客观性 | 双向审计主实验 | 不需要 |
| PeerQA | NAACL 2025；问题来自同行评审，答案由论文作者标注 | Paper-RAG 检索实验 | 不需要 |
| OpenReview ICLR/NeurIPS 子集 | 官方公开投稿、评审、回复和决策 | 候选生成和端到端案例 | 不需要 |
| 本地成对论文库 | 固定 PDF/Markdown 快照 | 受控 Literature-RAG | 不需要 |

### 8.2 辅助数据源

| 数据源 | 用途 | 使用边界 |
| --- | --- | --- |
| SubstanReview | 评审 claim-evidence 充分性辅助评价 | 不直接作为论文正文证据 gold |
| RottenReviews | factuality、vagueness、actionability 等报告质量评价 | LLM judge 只作辅助 |
| DeepReview-13K / DeepReview-Bench | 结构化评审和端到端对照参考 | 视授权和下载条件使用 |
| PRISM | 多维评审评价框架参考 | 作为前沿评价参考，不依赖其作为主数据 |

### 8.3 Literature-RAG 数据

受控 Literature-RAG 使用：

1. `参考论文/整合去重_论文库/PDF_MD_成对论文库/`；
2. 目标论文参考文献表中的真实文献；
3. OpenReview 真实评审中明确提及的相关工作或 baseline；
4. OpenAlex 和 Semantic Scholar 的元数据补全结果。

使用真实引用关系和明确 reviewer citation 构造 silver relevance，不新增大规模人工标签。Citation Validity 通过 DOI、arXiv ID、OpenAlex ID 或 Semantic Scholar Paper ID 自动核验。

### 8.4 数据划分与防泄漏

1. 按论文划分训练、验证和测试，禁止同一论文跨集合；
2. 同一论文不同版本只进入同一划分；
3. OpenReview 数据保存 venue、paper id、版本、抓取日期和 API 版本；
4. 主结果优先使用现成 gold 标签；
5. 模型生成标签只能作为 proxy，必须单独报告；
6. 端到端案例不能替代严格组件实验；
7. Literature-RAG 按投稿日期进行时间过滤，避免检索未来文献。

## 九、实验设计

### 9.1 实验总体原则

1. 先评价组件，再评价端到端系统；
2. 所有对比使用相同模型、候选上限和 token 预算；
3. 主实验至少重复三次，报告均值、标准差和 95% 置信区间；
4. 使用配对 bootstrap 或配对显著性检验；
5. 每项创新必须有对应 Baseline 和消融；
6. 所有实验保存 manifest、Prompt、模型配置、输入输出和错误日志。

### 9.3 E0：数据与协议准备

工作内容：

- 下载并审计 CLAIMCHECK、PeerQA、OpenReview 和辅助数据；
- 统一 paper、block、weakness、evidence 和 decision schema；
- 构建 Paper-RAG 和 Literature-RAG 索引；
- 固定实验划分和配置；
- 建立 manifest 与结果表生成脚本。

输出：数据注册表、解析质量报告、索引快照和实验配置。

### 9.4 E1：候选弱点生成诊断

对比 G0–G2，固定每篇论文候选数和 token 预算。

指标：

- Valid Candidate Precision；
- Weakness Coverage；
- Specificity；
- Generic Rate；
- Redundancy Rate；
- 平均候选数量；
- token、成本和延迟。

目的：选定稳定候选生成器，不用于证明核心创新。

### 9.5 E2：结构与证据类型感知 Paper-RAG

数据：PeerQA 与 CLAIMCHECK 可定位证据样本。
对比：P0–P4。

指标：

- Recall@3/5/10；
- MRR；
- nDCG@5；
- Section Match；
- Evidence-Type Match；
- 检索延迟。

消融：

- 去掉 Section Prior；
- 去掉 Evidence-Type Prior；
- 去掉 Neighbor Expansion；
- 仅 BM25；
- 仅 Dense。

### 9.6 E3：受控 Literature-RAG

数据：本地文献快照、真实参考文献和 OpenReview reviewer citation。
对比：L0–L3。

指标：

- Recall@5/10；
- MRR；
- Literature Relevance@K；
- Citation Validity Rate；
- Temporal Validity Rate；
- 检索延迟。

该实验控制为 20–30 个查询和固定候选池，属于次要实验，不进行 Web-scale 检索。

### 9.7 E4：双向证据审计主实验

数据：CLAIMCHECK 为主，SubstanReview 为辅助。
对比：A0–A4。

指标：

- Valid-Issue Precision；
- Covered/Refuted Recall；
- Adjudication Macro-F1；
- Evidence Attribution Accuracy；
- False-Keep Rate；
- `uncertain` Rate；
- 每条候选 token、成本和延迟。

关键对比：

- A2 vs A4：验证双向审计相对单 Judge 的价值；
- A3 vs A4：验证 Refutation Agent 的增量价值；
- A0 vs A4：验证生成后审计对错误保留的抑制作用。

### 9.8 E5：Meta-Reviewer Ranker

数据：同一论文内经过审计的候选弱点集合。
对比：K0–K4。

指标：

- Top-3 / Top-5 Precision；
- nDCG@5；
- Major Weakness Coverage；
- Redundancy Rate；
- Evidence Coverage；
- Confidence Calibration；
- 报告压缩率。

消融：

- 无 evidence_sufficiency；
- 无 literature_support；
- 无 redundancy_penalty；
- 无 uncertainty_penalty。

### 9.9 E6：端到端自动评审

在固定 OpenReview 小样本上运行五个端到端系统。

指标：

- Valid Weakness Precision；
- Evidence Coverage；
- Generic Rate；
- False-Keep Rate；
- Redundancy Rate；
- Citation Validity Rate；
- 成本、延迟与运行稳定性；
- RottenReviews / PRISM 风格的辅助多维质量分数。

输出 3–5 个完整案例，展示候选弱点、检索结果、支持案例、反驳案例、自动裁决和最终排序。

## 十、评价指标与实验成功标准

### 10.1 指标定义

| 层级 | 核心指标 | 解释 |
| --- | --- | --- |
| 检索层 | Recall@K、MRR、nDCG、Section Match、Evidence-Type Match | 是否找到正确且类型合适的证据 |
| 审计层 | Macro-F1、Valid-Issue Precision、Covered/Refuted Recall、False-Keep Rate | 是否正确保留或删除候选意见 |
| 排序层 | Top-K Precision、Major Weakness Coverage、Redundancy Rate | 是否把重要且有效的问题排在前面 |
| 报告层 | Evidence Coverage、Generic Rate、Citation Validity | 最终报告是否具体、可追溯 |
| 工程层 | latency、token、cost、failure rate | 系统是否可实际运行 |

### 10.2 主成功标准

| 实验 | 成功标准 |
| --- | --- |
| E2 Paper-RAG | P4 相对最强非结构 Baseline P0–P2 的 Recall@5 和 nDCG@5 显著提升；Recall@5 目标提升至少 5 个百分点；Evidence-Type Match 提升至少 10 个百分点；延迟不超过 P2 的 2 倍 |
| E3 Literature-RAG | Citation Validity Rate ≥ 95%；L3 的 Recall@10 或 MRR 显著优于 L0；受控语料 Recall@10 目标 ≥ 0.70 |
| E4 双向审计 | A4 相对 A2 的 Valid-Issue Precision 和 Covered/Refuted Recall 均提升至少 5 个百分点，或一项提升至少 8 个百分点且另一项非劣于 2 个百分点；False-Keep Rate 相对降低至少 20%；Evidence Attribution Accuracy 目标 ≥ 0.75；成本不超过 A2 的 2.5 倍 |
| E5 Meta-Reviewer | K4 相对 K1/K2 的 Top-5 Precision 提升至少 5 个百分点；Redundancy Rate 相对降低至少 20%；Major Weakness Coverage 非劣于 3 个百分点 |
| E6 端到端 | Full EviReview-Lite 相对 Direct LLM 和 Three Reviewer Agents 提高有效弱点精度与证据覆盖率，并降低 Generic、False-Keep 和重复率；最终报告不存在无法核验的外部引用 |

### 10.3 总体成功判定

1. E4 双向证据审计必须通过主成功标准；
2. E2、E3、E5 中至少再有一项通过主成功标准；
3. E6 至少证明完整系统在两项核心报告指标上显著优于生成 Baseline；
4. 若 E4 未通过，论文应收窄为结构感知检索与证据化排序研究；
5. 所有结论必须同时报告效果、成本和失败案例。

## 十一、可行性分析与工作量论证

### 11.1 数据可行性

CLAIMCHECK、PeerQA、SubstanReview、RottenReviews 和 OpenReview 均有公开论文、代码、数据或 API 入口；本地已整理 84 个 PDF 和 71 个 Markdown 文献文件，可直接用于受控 Literature-RAG。主实验不依赖新增人工标注。

### 11.2 技术可行性

系统所需技术均有成熟开源实现：

- 论文解析可使用 MinerU、GROBID 或已有 Markdown；
- BM25、Dense、RRF 和向量库均有稳定实现；
- Agent 使用固定状态机和结构化输出，不需要复杂自由讨论；
- LLM 使用兼容 API，不需要本地训练；
- 指标和统计检验可使用 Python 实现。

### 11.3 工作量符合硕士毕业设计要求

本课题包含：

1. 一套完整后端系统架构；
2. 8 个职责清晰的 Agent；
3. Paper-RAG 与 Literature-RAG 两套检索流程；
4. 统一审计数据协议与可追踪运行记录；
5. 5 类 Baseline、7 个实验阶段及主要消融；
6. 数据处理、服务接口、测试、实验和论文表格生成；
7. 误差分析、成本分析和案例研究。

同时通过不训练模型、不做 Web-scale 检索、不执行论文代码和不新增大规模人工标注控制风险。因此，工作量足以支撑工程型硕士毕业设计，又不会扩展为难以完成的大型研究项目。

| PDF 解析错误 | 证据定位不准 | 优先使用 Markdown/LaTeX；保留章节与邻近块 |
| CLAIMCHECK 标签映射不完全 | 审计实验偏差 | 预先固定映射表，单列不可映射样本 |
| LLM 输出不稳定 | 实验方差较大 | 固定模型、Prompt、温度和预算，重复运行 |
| LLM judge 偏差 | 最终评价失真 | 主结果使用 gold；Judge 只作辅助 |
| Literature-RAG 引用错误 | 新颖性意见不可靠 | 强制 ID/URL/片段；计算 Citation Validity |
| 双向审计成本较高 | 难以扩展 | 候选上限、缓存、共享索引、批量推理 |
| 自动评审误用 | 影响真实审稿 | 明确定位为投稿前自检和评审辅助，不输出录用决定 |

系统不应对作者身份、机构或匿名评审人进行推断，不应抓取非公开评审数据，也不应将自动评审结果直接作为正式录用决定。
