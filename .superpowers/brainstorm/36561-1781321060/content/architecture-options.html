研究原型的三种架构选择

目标：单机可复现、组件可消融、完整审计轨迹、避免过度工程。点击你认可的方案。

Controller / API
Model：论文、评审、实验数据
View：报告与界面

A. MVC 单体

适合普通 CRUD，但 Agent 流程、检索实验和消融边界会混入 Controller/Model，不适合作为核心架构。

API / CLI
Application Services
Domain Logic
Repositories / Providers

B. 传统分层模块化单体

简单易懂，但实验变体容易通过条件分支侵入业务服务，长期会降低消融清晰度。

Entrypoints：CLI / API / Experiment Runner
Use Cases：审计评审 / 检索 / 分类
Domain：Comment / Evidence / Verdict / Trace
Ports：LLM / Retriever / Store / Evaluator
Adapters：GLM / Qdrant / BM25 / Dataset

C. 研究型六边形模块化单体（推荐）

核心协议稳定，模型、检索器和数据集通过适配器替换；天然支持 baseline、消融与离线复现。

推荐 C 的原因

  • 每个 Agent 节点可单测、替换和消融
  • 实验 Runner 与 API 共用同一核心逻辑
  • 本地 BM25 与 Qdrant、Mock LLM 与真实模型可互换
  • 审计轨迹作为一等数据对象

明确不做

  • 不拆微服务
  • 不引入 Kubernetes
  • 不让 MVC 控制核心 Agent 流程
  • 不使用自由协商式 Agent 群聊
